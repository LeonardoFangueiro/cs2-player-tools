01 Valve Infrastructure & Steam Datagram Relay

CS2 usa o Steam Datagram Relay (SDR) — uma rede de relays proprietária da Valve que encaminha todo o tráfego de jogo. Os jogadores nunca se conectam diretamente aos game servers em matchmaking.

Proteção DDoS

IPs dos game servers nunca são expostos aos jogadores.

Routing Otimizado

Tráfego usa o backbone privado da Valve em vez de depender do peering dos ISPs.

NAT Traversal

Elimina problemas de port-forwarding completamente.

Privacidade

IPs reais dos jogadores ficam ocultos de servers e outros players.

Como funciona o SDR

A arquitetura SDR é composta por 3 componentes:

  1. Relay PoPs (Points of Presence) — Nós edge distribuídos globalmente. Lightweight, encaminham pacotes UDP encriptados sem inspecionar conteúdo
  2. Valve Backbone — Links de rede privados entre datacenters da Valve
  3. Game Server DC Relays — Endpoints no mesmo datacenter que os game servers ("exit nodes")
Player <--UDP--> Relay PoP (mais próximo) <--Valve backbone--> DC Relay <--local--> Game Server

Segurança

  • Encriptação end-to-end: AES-256-GCM entre client e game server. Relays NÃO conseguem ler dados do jogo
  • Autenticação: Via certificados assinados pela CA root da Valve
  • Connection tickets: Assinados pela chave privada da Valve
  • Key exchange: Curve25519 ECDH

Fluxo de Conexão Completo

  1. Startup do Client: CS2 carrega o SDR network config via Steam API. Contém todos os relay PoPs, IPs, portas e chaves públicas criptográficas.
  2. Medição de Latência: Probes UDP lightweight a TODOS os relay PoPs. RTTs reportados ao matchmaking.
  3. Queue para Match: Game Coordinator (GC) recebe pedido + ping data a todos os clusters.
  4. Match Encontrado: GC seleciona game server que minimiza ping agregado para os 10 jogadores.
  5. Connection Ticket: Ticket assinado criptograficamente emitido a cada jogador.
  6. Túnel SDR: Client → relay PoP (menor latência) → túnel ao DC do game server via backbone Valve.
  7. Sessão: Todo o tráfego flui: Player ↔ Relay PoP ↔ Valve backbone ↔ DC Relay ↔ Game Server
  8. Relay Failover: Se relay degradar, switch transparente para o próximo melhor sem desconexão.
Seleção de Relay: O client minimiza client_to_relay_ping + relay_to_server_ping. SDR pode usar multi-path simultaneamente e selecionar o melhor.

02 CS2 Network Protocol & Sub-tick

Protocolo: UDP exclusivamente para game traffic. Encapsulado dentro do SteamNetworkingSockets (reliability layer própria + AES-256-GCM + sequencing sobre raw UDP).

Sistema Sub-tick

CS:GO usava 64 tick (MM) / 128 tick (FACEIT). CS2 introduziu o sub-tick: servidor continua a 64 ticks/segundo mas inputs têm precisão dentro do tick.

Como Funciona

  1. Client amostra inputs a alta frequência (ligado ao framerate, 200-500+ Hz)
  2. Cada input é timestamped com precisão sub-tick
  3. Client envia usercmd com número do tick + fração sub-tick (0.0 a 1.0)
  4. Server usa timestamp sub-tick para interpolar posições para hit detection
  5. Para disparos: server rewind ao momento exato sub-tick, avalia hit, produz resultado
Impacto na rede: A rate de envio base continua a 64Hz. Dados sub-tick são metadata dentro dos pacotes existentes. Jogadores com framerate mais alto obtêm timestamps mais precisos.

Bandwidth Típico

  • Downstream (server → client): ~80-120 KB/s (640-960 kbps)
  • Upstream (client → server): ~20-40 KB/s (160-320 kbps)
  • Snapshots são delta-compressed — só mudanças desde o último estado acknowledged

Lag Compensation

CS2 usa server-side lag compensation (backward reconciliation):

  1. Jogador dispara → client regista tempo exato (sub-tick) e view do mundo
  2. Server recebe shot command → rewind game state ao momento do disparo
  3. Server avalia disparo contra posições rewound de todos os jogadores
  4. Se acertou no estado rewound → hit registado
Trade-off: Um jogador alvo pode ser atingido mesmo que no seu ecrã já se tenha movido para trás de cobertura ("peeker's advantage" / "dying behind walls"). Janela máxima: ~200ms.

Packet Loss Handling

  • Reliability selectiva: Dados críticos → reliable com retransmissão. Position updates → unreliable (próximo update substitui)
  • Interpolação: Buffer suaviza snapshots em falta. cl_interp_ratio 2 = buffer extra
  • Jitter buffer: Pacotes ligeiramente atrasados ainda usáveis se dentro da janela de interpolação

Interpretar o cq_netgraph

  • Ping (ms): <50ms excelente, 50-80 bom, 80-120 jogável, 120+ problemático
  • Loss (%): 1-2% já causa problemas de hitreg. >5% severo
  • Server Recv Margin: Valores negativos (vermelho) = dados chegam tarde demais. Métrica mais crítica
  • Choke: Updates não processados a tempo = bottleneck CPU client

Ferramentas Externas

PingPlotter (Recomendado)

  1. Instalar PingPlotter (free version funciona)
  2. Target: IP do relay PoP de net_connections_status
  3. Correr durante match ou mais
  4. Procurar: latency spikes em hops, packet loss intermédio, latência alta consistente
ICMP deprioritization: Loss num hop intermédio que NÃO persiste nos seguintes = rate-limiting de ICMP (ignorar). Só loss no último hop é definitiva.

Distinguir Routing vs Server Issues

Problema de Routing

  • PingPlotter mostra loss começando num hop específico
  • Outros jogadores não afetados
  • Consistente para regiões específicas
  • VPN resolve o problema

Problema Server-Side

  • Múltiplos jogadores afetados simultaneamente
  • Server var spikes no netgraph
  • Intermitente, sem correlação ISP
  • Steam status mostra problemas

Problemas Comuns

PadrãoCausa Provável

Buffer Bloat

O maior causador de latency spikes sub-diagnosticado. Testar em waveform.com/tools/bufferbloat ou speed.cloudflare.com.

Fix: Ativar SQM/fq_codel no router. Bandwidth limits a ~85-90% das velocidades reais.

  • Interrupt Moderation Rate: Off

WFP / Antivirus

Kaspersky, Bitdefender, ESET com firewall module instalam WFP callout drivers que adicionam latência. Adicionar CS2 como exclusão.

Desativar Features Desnecessárias

  • IPv6 (se ISP não necessitar): Disable-NetAdapterBinding -Name "Ethernet" -ComponentID ms_tcpip6
  • NetBIOS: Adapter Properties → IPv4 → Advanced → WINS → Disable
  • Link-Layer Topology Discovery: Desmarcar ambos drivers
  • QoS Packet Scheduler: Manter ENABLED (mito dos 20% é falso)

05 WireGuard VPN para Gaming

Porquê WireGuard

AspetoWireGuardOpenVPN UDPOpenVPN TCPIPSec
Overhead/pacote60 bytes70-10070-100+70-90
Latência extra~0.5-1.5ms~2-5ms~5-15ms+~1-2ms

TransporteUDP onlyUDPTCP (meltdown!)UDP

JitterMuito baixoVariávelAltoBaixo

Criptografia: Curve25519 (key exchange) + ChaCha20-Poly1305 (AEAD) + BLAKE2s (hashing). Rekeying a cada 2 min com zero interruption.

O Problema: Bad ISP Peering

BGP otimiza para custo/política, NÃO latência. Exemplo:

Sem VPN: Lisboa → Londres → Frankfurt → Madrid  (+30-50ms)
Com VPN:  Lisboa → VPN Madrid (5-15ms) → Valve Madrid (1-2ms) = ~7-17ms

Split Tunneling para CS2

AllowedIPs do WireGuard define que IPs vão pelo túnel. Apenas tráfego Valve passa pelo VPN:

[Peer]
PublicKey = <server-public-key>
Endpoint = <vps-ip>:51820
PersistentKeepalive = 25

# Split tunnel: apenas IPs Valve (AS32590)
AllowedIPs = 155.133.224.0/19, 162.254.192.0/21, 208.64.200.0/21, 185.25.180.0/22, 192.69.96.0/22, 205.196.6.0/24, 103.10.124.0/23, 103.28.54.0/23, 146.66.152.0/21, 208.78.164.0/22
Dinâmico: IPs de relay SDR mudam. A app deve fazer fetch da SDR config API e atualizar AllowedIPs automaticamente com wg set.

Interação VPN + SDR

  • Com VPN, a localização aparente muda para a do VPN server
  • SDR mede latência desde o VPN exit point, seleciona relay mais próximo daí
  • VPN ajuda se o server está perto de um Valve PoP e o ISP path ao VPN é melhor que ao PoP direto

Cenário Benéfico

Tu (PT) → VPN (Frankfurt) → SDR Relay Frankfurt (~1ms) → Game Server Luxembourg (~3ms)

Total: teu_ping_ao_VPN + 4ms

Cenário Prejudicial

Tu (PT) → VPN (US East) → SDR Relay US East → Game Server Luxembourg

+80-100ms desnecessários

VAC e VPN

VPN NÃO é banido pelo VAC. Valve nunca proibiu uso de VPN. VAC inspeciona memória e processos, não routing. Steam Guard pode pedir verificação ao login de IP novo — inconveniente, não risco.

Implementação Windows

Componentes

  • wireguard.exe — GUI + tunnel service manager
  • wg.exe — CLI tool
  • wintun.dll — High-performance Layer 3 TUN driver
  • Service: WireGuardTunnel$<name>

Controlo Programático

# Instalar tunnel como Windows service
wireguard.exe /installtunnelservice "C:\path\to\tunnel.conf"

# Remover tunnel
wireguard.exe /uninstalltunnelservice MyTunnel

# Gerir via services
sc start WireGuardTunnel$MyTunnel
sc stop WireGuardTunnel$MyTunnel

# wg.exe CLI
wg show MyTunnel                   # Status
wg show MyTunnel dump              # Machine-readable
wg set MyTunnel peer <key> allowed-ips 10.0.0.0/24
wg genkey | wg pubkey              # Key pair

Named Pipe (UAPI) — Acesso Programático

\\.\pipe\ProtectedPrefix\Administrators\WireGuard\<tunnel-name>
# Protocolo text-based: get=1, set=1

wireguard-nt — Kernel Driver para Embedding

C API (WireGuardCreateAdapter, WireGuardSetConfiguration) para máxima integração. GitHub: WireGuard/wireguard-nt.

06 Feature Roadmap

Tool 1: Network Diagnostics Dashboard ALTA

Real-time ping a todos os Valve PoPs, traceroute visual, detecção automática de loss/jitter/routing issues, histórico por sessão, alerta de degradação, teste buffer bloat.

Tool 2: Windows Network Optimizer ALTA

Scan de settings atuais, one-click optimization com backup/restore, perfis Gaming/Normal/Custom, detecção NIC (Intel/Realtek), alerta de software interferente, MTU auto-detection, DNS config.

Tool 3: Smart VPN (WireGuard) ALTA

Profile management, auto-connect com CS2, split tunnel inteligente (fetch dinâmico IPs Valve), latency testing, server recommendation, real-time stats, one-click VPS deploy via cloud APIs.

Tool 4: Server Picker / Region Lock MÉDIA

Mapa mundial interativo com PoPs e latências, bloquear/desbloquear regiões via Firewall rules, sugestão automática melhores regiões.

Tool 5: Connection Monitor (Live Overlay) MÉDIA-BAIXA

Overlay in-game com ping real, loss/jitter, relay path, VPN status. Log de sessão. Game State Integration (GSI).

Tech Stack Recomendada

Recomendação: Tauri v2 + Rust + React

  • Tauri v2: Framework desktop (~10-15MB vs ~200MB Electron)
  • Rust backend: Win32 APIs via windows-rs, performance nativa, segurança memória
  • React frontend: UI moderna, dashboards, gráficos (Chart.js/Recharts), mapa interativo
  • WireGuard: Via CLI ou bindings Rust / wireguard-nt
  • System access: Registry, adapters, services, firewall — tudo via Rust
  • Packaging: MSI installer nativo
